## 核心思路

发布方案里有两类内容，处理方式不同：

| 类型 | 特征 | 处理方式 |
|------|------|----------|
| **结构化事实** | 有唯一正确答案：git 地址、分支、PR 状态、配置文件路径 | **机器采集**，从代码仓库和协作平台读取 |
| **决策判断** | 需要拍板：何时发、怎么灰度、如何回滚、风险怎么处置 | **人工撰写**，AI 可出草稿但不直接定稿 |

**原则：机器填表，人写决策。**

手工慢，不是因为打字慢，而是因为**信息散落在 git、PR、代码目录、Wiki、IM 里**，人要逐个查找再粘贴。自动化的价值是把「找和贴」变成「读和填」。

## 端到端流程（思路）

### 阶段 0：准备 — 身份与代码就绪

两件事先到位，后面才能跑通：

1. **协作身份就绪**  
   需要往 Wiki 写文档时，操作者以**本人飞书身份**登录工具链。读代码、生成草稿不依赖飞书；**只有发布和更新 Wiki 时才需要登录**。

2. **业务代码在本地可读**  
   相关仓库已在开发机 clone，且 git / 代码托管平台（如 GitHub）可正常访问。自动化**不会远程拉代码**，只读本地已有仓库。

---

### 阶段 1：自动采集 — 从代码和平台读「事实」

以 Wiki 官方「需求发布计划」模板为骨架，从以下来源**自动填入结构化字段**：

| 信息来源 | 读什么 | 填到发布方案哪里 |
|----------|--------|------------------|
| **Git 仓库** | 远程地址、当前分支 | 发布前检查 · 仓库表 |
| **代码托管 PR** | CR 链接、review 状态、是否合入主分支 | 发布前检查 · 仓库表 |
| **代码目录** | yml、nacos、kafka、DB 迁移脚本等配置是否存在 | 发布项 check list |
| **外部链接（传入）** | PRD、技术方案 Wiki、测试报告 | 发布概述 · 相关链接 |
| **服务清单（传入）** | 本次发布涉及的服务名 | 代码发布 · 服务表 |

采集结果是**草稿**：能确定的填上，确定不了的标「待补充」，**不做业务决策**。

配置扫描目前是**存在性探测**（有没有改 yml、有没有 migration 脚本），不是完整 diff 解读；网关、定时任务、环境变量等仍需人工核对。

---

### 阶段 2：人工决策 — 补「无法自动推断」的内容

脚本生成草稿后，负责人必须人工完成以下段落：

1. **发布概述** — 这次发什么、何时发、影响谁（简短、说人话）
2. **外部依赖** — 依赖哪个团队、对方何时 ready；没有就写「无」
3. **灰度方案** — 是否灰度、谁执行、切量比例和时间
4. **回滚方案** — 回滚版本、配置备份、执行人、预计耗时
5. **系统风险** — 具体风险 + 出问题时做什么（不要空话）

同时补全脚本填不了的字段：上线 tag、代码扫描结果、CR 负责人、发布风险等级、步骤说明等。

---

### 阶段 3：质量门禁 — 发布前校验

草稿定稿前过一道**格式与内容门禁**：

- 章节、表格列数、checkbox 文案是否与官方模板一致
- 概述是否过长、是否出现空话套话
- check list 每项是否有结论（无变更写「无」）

不通过则回到阶段 2 修改；通过才允许挂 Wiki。

---

### 阶段 4：发布协作 — 挂 Wiki 给评审

校验通过的草稿，以操作者飞书身份**创建文档并挂到「发布文档」节点下**，生成链接供：

- 产品 / 运维 / 发布窗口评审
- 发布执行时对照 check list 和发布顺序

本地保留 md 副本，便于 diff 和版本追溯。

---

### 阶段 5：迭代 — 评审后局部更新

评审有反馈时，**只改对应段落或表格行**，不整篇重写。避免：

- 丢失 Wiki 评论和协作记录
- 格式被意外改乱
- 历史发布计划无法对比

---

## 流程全景

```
准备（身份 + 本地代码）
    ↓
自动采集（git / PR / 配置 / 链接 → 模板草稿）
    ↓
人工决策（概述 / 依赖 / 灰度 / 回滚 / 风险）
    ↓
质量门禁（格式 + 说人话）
    ↓
挂 Wiki「发布文档」→ 评审
    ↓
局部更新（有反馈时）
```

## 填什么 · 谁填

| 区块 | 填什么 | 谁填 |
|------|--------|------|
| **发布概述** | 背景 + PRD / 技术方案 / 测试链接 | 人写，链接可自动带 |
| **外部依赖** | 依赖团队、对方完成时间 | 人写 |
| **发布项 check list** | yml / nacos / kafka / DB / 网关 / xxl-job / ENV / 扩容 | 机器扫代码，人核对 |
| **发布前检查** | 9 列仓库表 | 机器采 git / PR，人补扫描结果 |
| **前置准备** | 10 项 checkbox | 人逐项确认勾选 |
| **代码发布** | 11 列服务表 | 机器填骨架，人补 tag / 风险 |
| **发布后验证** | 验证步骤 + 监控 + 测试报告链接 | 人写，测试报告可挂链 |
| **灰度 / 回滚 / 风险** | 方案与动作 | 人写定稿 |

---

## 填写规则（3 条）

1. **只填内容，不改格式** — 章节顺序、表格列名、checkbox 文案与官方模板一致
2. **表格机器填，段落人来说** — 概述 / 灰度 / 回滚 / 风险必须人工定稿
3. **无变更写「无」** — check list 不留空

---

## 耗时对比（单次，3 仓库 + 2 服务 · 估算）

| 环节 | 纯手工 | 按流程 |
|------|--------|--------|
| 表格 + 配置 + 链接 | ~70 min | ~5 min |
| 概述 / 灰度 / 回滚 / 风险 | ~30 min | ~15 min |
| 格式返工 | ~15 min | ~2 min |
| **合计** | **~115 min** | **~22 min** |

纯手工时间主要耗在**跨系统找信息**（git、PR、配置、Wiki），不是写决策段落本身。真实耗时应通过分阶段计时表实测校准。

## 当前边界（思路层面）

| 已覆盖 | 尚未覆盖 |
|--------|----------|
| 从本地代码读 git / 配置事实 | 自动 clone 远程仓库 |
| 从 PR 平台读 review / 合入状态 | 完整配置 diff 解读 |
| 模板格式校验 | CI 触发自动生成草稿 |
| 挂 Wiki 发布 | 耗时自动统计 |

测试报告（如 Newman）可作为「发布后验证」的附件链接，与填发布方案是**可选并行**步骤。

---

## 三层分工

| 层级 | 定位 | 受众 |
|------|------|------|
| **Wiki 模板** | 发布方案长什么样 | 所有填表人 |
| **Wiki 本文档** | 为什么这样填、端到端思路 | 评审 / 负责人 |
| **代码仓库 + 工具** | 具体脚本与实现 | 开发 / Agent 执行 |

Wiki 讲思路和规则；具体命令与脚本源码在代码仓库维护，不在本文展开。
